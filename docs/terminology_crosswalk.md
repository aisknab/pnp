# Terminology Crosswalk

## Purpose and scope

This document translates the claim-critical terminology used by the canonical report and the proof/checker source into conventional complexity-theory, proof-engineering, formal-methods, and software-assurance language.

It is descriptive, not an endorsement. A definition recorded here states what the repository means by a term; it does not establish that the corresponding theorem or checker rule is correct.

The release-specific source coordinate used for this crosswalk is:

```text
source tag:    final-pnp-proof-report-hardened-7072f8d
source commit: 7072f8d0bda6d44d240f9bb3fad624fd357e1278
```

Coverage includes:

- every claim-critical bespoke term named in the review brief;
- the mathematical objects required to understand those terms;
- every package-family name in the report's theorem ledger and in `LOCAL_THEOREM_BY_FAMILY0`;
- release and checker terms whose repository-specific meaning differs from ordinary usage.

Ordinary programming words are omitted where the repository uses their normal meaning. Record-field names are grouped under their owning concept rather than treated as independent mathematical definitions.

### How to read the tables

- **Formal definition** is the repository definition or the narrowest faithful normalization of it.
- **Closest standard concept** says â€śno exact standard equivalentâ€ť when the term is genuinely project-specific.
- **Minimal example** is intentionally small and usually conceptual. It is not proof evidence. Runnable examples are a later documentation step.
- **Checker module/function** identifies where enforcement is implemented or where the top-level acceptance chain checks the relevant record. A function accepting a record does not by itself establish that the encoded mathematical rule is sound.

## Ambiguity and naming rules

1. **`Residual` must be qualified.** Use â€śresidual slack,â€ť â€śpositive residual witness,â€ť â€śBN4 residual cell,â€ť or â€śquotient residual key.â€ť
2. **`Certificate` is not synonymous with proof.** State the checker, schema, conclusion, and trust assumptions.
3. **`Acceptance` must name its scope.** Prefer `CheckGPack0 accepted`, `CheckPCCPackexp0 accepted`, or `CheckFinalPNPProofReport0 accepted` over â€śthe proof was accepted.â€ť
4. **`Seal` means identity, not correctness.** A SHA-256 match establishes that bytes match a manifest.
5. **`Firewall` means a checked proof boundary.** It is not a network-security control.
6. **`Minimization trace` is not a canonical single object.** Name the actual normalization trace, descent sequence, acceptance run, or replay.
7. **Package letters are namespaces, not theorem names.** Cite the reflected theorem ID and checker function together.
